# PortOSC AI 工作流（重构版）

## 目的
在一次性大范围重构场景下，保证改造完整性、连续性和可回退性，用于“重构 + 新功能 + Bug 修复”。

## 强约束（本项目当前规则）
1. **实施方式**：按一次性整体重构执行，不拆分为多个小步提交。
2. **提交策略**：每次修改完成后仅进行本地 `git commit`，禁止 `push`。
3. **版本管理**：采用三段式版本号 `1.2.3`，并按本项目定义执行。
4. **验证要求**：完成后必须构建验证，并记录关键功能回归结果。

## 一次性重构执行流程
1. **任务冻结**：明确本轮要同时完成的重构目标、功能和修复清单。
2. **现状建模**：整理当前架构（UI、通信、解析、绘图、配置、日志）及耦合点。
3. **目标架构设计**：给出重构后分层、职责边界、关键接口和数据流。
4. **全量实现**：按目标架构直接完成改造（允许跨文件/跨模块联动）。
5. **全量回归**：执行构建与关键路径回归（串口/TCP/UDP、收发、曲线、配置）。
6. **版本落档**：更新版本记录并完成本地提交。

## 版本策略（项目定义）
- 使用 `1.2.3` 三段式：
  - `1`：大版本（通常不改，只有根本性功能/结构重构时才调整）
  - `2`：可推送小版本（完成一组修改与验证后，作为可推送到 GitHub 的版本）
  - `3`：修订号（每次本地 `git commit` 后必须 `+1`）
- 版本推进示例：
  - 本地连续开发：`1.1.0 -> 1.1.1 -> 1.1.2`
  - 达到可推送里程碑并推送后，进入下一轮开发：`1.2.0 -> 1.2.1`

## 本轮重构目标（已追加）
1. **逻辑重构**：重写高频场景下曲线更新逻辑，解除“数据解析 + UI线程”深耦合导致的显示不稳定。
2. **接收端口模块化**：将接收端口逻辑从 `Form1` 进一步解耦，形成可扩展通信协议接入模型。
3. 保持现有可用功能连续性（串口/TCP/UDP、转发、配置、发送、显示）。
