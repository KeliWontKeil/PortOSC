# PortOSC 开发与重构工作流

## 1. 文档目的
本文档用于约束后续所有开发与重构任务的执行方式，目标是：
- 保持现有可用功能连续性
- 降低 `Form1` 与传输层、解析层的耦合
- 保证每次改动都可验证、可回退、可追踪
- 为后续新增协议、解析规则、绘图能力提供统一流程

## 2. 适用范围
本工作流适用于以下任务：
- 功能新增
- Bug 修复
- 结构重构
- 性能优化
- 协议接入扩展
- 配置格式调整
- UI 行为调整

不适用于纯文案修订、纯注释调整、格式化整理等低风险变更；这类改动可直接执行，但仍建议保留最小验证。

## 3. 项目执行原则
### 3.1 任务原则
1. 先明确目标，再动代码。
2. 优先保持行为不变，再做结构改进。
3. 变更范围尽量聚焦，但允许跨文件联动完成一个闭环。
4. 每次改动都要回答两个问题：
   - 这次改了什么？
   - 如何确认没有破坏已有功能？
5. 当本轮任务属于重构或解耦时，应在同一轮内尽可能连续推进多个可验证子阶段，不要在每个小阶段停下来等待确认。
6. 只有在以下情况才主动暂停并汇报：
   - 需要运行时功能验证或人工观察结果
   - 发生构建失败或行为偏差
   - 已完成一批足够大的重构任务，需要统一总结

### 3.2 代码原则
1. 复用现有工具与约定，避免重复造轮子。
2. 新增抽象必须有明确用途，不能为了“看起来更整洁”而引入。
3. UI 只能负责展示与交互，不应承担协议解析、生命周期协调等核心业务。
4. 传输层、解析层、绘图层、配置层应尽量单向依赖，避免互相调用。

### 3.3 变更原则
1. 先保功能，再提结构。
2. 先做低风险收敛，再做高风险拆分。
3. 涉及行为变化时必须同步更新文档。
4. 任何影响接收、发送、绘图、配置的改动，都必须做回归验证。

## 4. 项目当前关键事实
### 4.1 当前技术栈
- `.NET 8`
- `Windows Forms`
- C# 12

### 4.2 当前架构特征
- `Form1` 仍承担大量编排职责
- 串口 / TCP / UDP 通过事件将数据抛给 UI
- 接收管线已开始拆分，但还未完全模块化
- 高频绘图依赖定时刷新与队列策略，属于重点关注区域

### 4.3 当前高风险点
- 控件文本被当作业务状态判断依据
- 状态切换代码分散且重复
- 接收、转发、显示、绘图混在一个流程中
- 异常处理不统一
- 配置解析缺少强校验

## 5. 每次任务的标准流程
### 5.1 任务冻结
开始前必须明确以下内容：
- 本次任务目标
- 影响范围
- 不允许改变的行为
- 验收标准
- 是否需要同步文档或测试

输出要求：
- 用一句话概括目标
- 列出涉及文件或模块
- 列出预期回归点

### 5.2 现状建模
先确认当前实现，再决定修改路径。重点看：
- 数据从哪里来
- 数据如何进入 UI
- 哪些地方负责状态切换
- 哪些地方负责错误处理
- 哪些地方存在重复逻辑

输出要求：
- 识别主要依赖链
- 标记耦合点
- 标记重复代码
- 标记高频路径

### 5.3 目标设计
在动代码前先明确目标结构：
- 模块职责边界
- 数据流方向
- 状态模型
- 错误上报方式
- 资源释放边界

输出要求：
- 描述“改造后应该长什么样”
- 明确哪些文件会改
- 明确哪些文件不应被动

### 5.4 实现阶段
实现时遵循以下顺序：
1. 先抽公共逻辑
2. 再替换调用点
3. 再处理兼容层
4. 最后清理冗余代码

实现要求：
- 尽量小步落地，但一个任务必须闭环
- 不引入无验证的新路径
- 保留失败回退点

### 5.5 验证阶段
每次完成后至少做以下验证：
- 构建是否通过
- 核心页面是否能正常打开
- 串口 / TCP / UDP 是否仍可连接
- 发送和接收是否正常
- 曲线是否仍可刷新
- 配置是否仍可加载和保存

如果当前变更仍处于重构链路中且仅涉及编译可验证的代码改动，则在构建通过后应自动继续下一个计划步骤，不必为每个子阶段单独暂停。

如任务涉及解析逻辑，还需补充：
- 输入格式边界
- 空输入
- 非法输入
- 粘包 / 半包或分片场景

### 5.6 版本记录
每次本地完成后必须记录：
- 当前版本号
- 本次改动概要
- 受影响模块
- 构建或回归结果

## 6. 版本策略
采用三段式版本号 `1.2.3`：
- `1`：大版本，通常只在架构级重构或重大功能变化时调整
- `2`：阶段版本，代表一组完成并验证过的变更
- `3`：修订号，每次本地提交递增

### 版本推进规则
1. 先确定当前基线版本
2. 完成一轮可验证变更后，再递增修订号
3. 只有在阶段性目标达成且回归通过后，才考虑提升中间位

### 版本推进示例
- `1.1.0 -> 1.1.1 -> 1.1.2`
- 达到可推送里程碑后，再进入下一轮 `1.2.x`

## 7. 任务分级规则
### 7.1 低风险任务
- 文档调整
- 常量整理
- 重复代码收敛
- 明确错误信息
- 小范围命名优化

### 7.2 中风险任务
- 解析管线拆分
- 配置读写重构
- UI 状态统一
- 发送分发收敛

### 7.3 高风险任务
- 统一传输层抽象
- 统一接收生命周期
- 绘图刷新模型重构
- 大范围协议适配

高风险任务必须先设计，再实现，再回归，不允许直接大面积替换。

## 8. 本项目推荐的重构顺序
1. 收敛 `Form1` 中重复的状态切换逻辑
2. 让发送入口脱离 UI 文本判断
3. 规范配置解析与校验
4. 完善接收管线并补测试
5. 优化高频绘图队列与刷新机制
6. 统一串口 / TCP / UDP 的生命周期与错误上报
7. 再考虑更大范围的协议抽象

## 9. 提交前检查清单
每次完成任务前，至少确认：
- 代码能编译
- 新增逻辑没有破坏原有入口
- 没有留下临时调试代码
- 没有吞掉异常
- 没有引入明显重复逻辑
- 文档已同步更新

## 10. 任务记录模板
建议每次任务都按以下格式记录：

- 问题与目标：
- 当前版本：
- 涉及模块：
- 目标架构变化：
- 修改文件：
- 风险等级：
- 验证结果：
- 未解决事项：
- 本地提交信息：

## 11. 给后续开发的执行说明
以后如果用户要求我做开发或重构，我应按如下顺序执行：
1. 先判断任务属于功能、修复还是重构
2. 先审查相关文件与现有工作流
3. 必要时先形成计划
4. 按目标设计执行修改
5. 完成后立即构建验证
6. 输出问题、改动、风险和后续建议

## 12. 本轮重构关注点
当前最重要的两个方向是：
1. 高频场景下曲线更新的稳定性
2. 接收端口逻辑从 `Form1` 中继续解耦

在这两个方向未完成前，不建议大规模扩展新功能。
